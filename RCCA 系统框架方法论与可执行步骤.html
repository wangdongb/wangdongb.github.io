① RCCA 是什么 · 为什么 · 什么时候用

1.1 概念区分

缩写全称含义
RCARoot Cause Analysis根本原因分析 — 找到问题真正的源头
RCCARoot Cause Corrective Action根本原因纠正行动 — 找到根因 + 制定纠正与预防措施 + 闭环验证

RCCA 比 RCA 多了"纠正措施 + 实施 + 跟踪"的闭环，是一种从问题入手、找到根因、制定措施、防止再发的过程方法。

1.2 为什么要做 RCCA

核心目标：高效解决复杂、重要、跨部门的问题，并杜绝重复发生。
  • 消除症状只让问题更难监控；消除直接原因只能短期缓解；唯有消除根原因才能彻底解决。
  • RCCA 是组织持续改进的核心活动，是一种"打硬仗"式的问题解决态度。

1.3 哪些问题需要走 RCCA

类型处理方式
单模块、原因明显的简单问题直接修复，不必走完整 RCCA
重要 / 系统集成 / 跨部门 / 长期或频繁出现 / 无法直接判断原因✅ 走 RCCA
问题会自然消失或解决成本远大于收益不必费周折

1.4 常见的分析失败原因（避坑清单）

⚠️ 常见失败
  • 信息收集不全面 / 问题定义不准确
  • 停在表面原因，没继续深挖
  • 视野狭隘，只关注单方面原因
  • 过早下结论，先入为主
  • 受经验影响沿用旧方案
  • 受参与人员知识范畴限制，没有正确的验证方向
  • 过多关注问题责任人指定，而非根因本身
  • 没有验证成功的方法 — 措施有效性未被证实

0

立项准备 — 成立分析小组

⏱ 半天 · 👥 1 人发起 → 全员到位
🎯 行动
  1. 判断问题是否符合 RCCA 启动条件（重要 + 复杂 + 跨部门 + 无法直接判断）。
  2. 申请管理层授权 — 没有授权很难推动跨部门执行。
  3. 确定组织者（一般是问题最先发现方或质量负责人）。
  4. 按业务模块组建小组，明确每位成员的职责和主管。
  5. 确定首次会议时间，建立沟通机制（群组、文档库、Jira tag）。
🧰 工具

团队矩阵表 + 利益相关者分析（区分核心利益相关者与一般利益相关者）。

📋 模板：分析小组矩阵
角色负责人主管主要职责
组织者 / Owner整体推进、会议主持、对结论负责
测试问题复现、对比测试、数据收集
硬件电子元器件、PCBA、整机硬件分析
结构结构件、装配、外观、材料分析
软件（上位机）App、固件交互、UI 逻辑
软件（下位机）MCU、控制算法、底层驱动
生产现场工艺、来料、产线 ATE/ICT 数据
供应商接口（按需）对接外部供应商问题反馈
⚠️ 常见陷阱
  • 组员只挂名，没有真正投入时间 → 提前与主管确认工时承诺。
  • 遗漏关键模块（如供应商、生产）→ 用流程图或鱼骨图先扫一遍涉及的"人机物法环"。
  • 没有授权 → 小组无权调动资源，会议成空谈。
1

定义问题 — 4W1H1S + HTDQ + 框架四原则

⏱ 1~2 天 · 这一步往往决定整个 RCCA 的成败
🎯 关键认知："问题定义不准确"是 RCCA 失败的头号原因。问题陈述模糊，后续所有工作都会偏离方向。这一步要花充足时间，宁可慢，不要错。
🎯 行动
  1. 用 4W1H1S 模板填充问题事实，事实必须可追溯（Jira ID、日志、复现视频等）。
  2. 用 HTDQ 框架（英雄-宝藏-龙-任务）把问题转化为一个"如何"开头的开放式任务。
  3. 用 框架完善四原则（背包、兔子、多利、华生）逐一检查问题陈述是否到位。
  4. 把问题陈述发给非小组成员看一眼，确认陌生人也能理解。
🧰 工具：4W1H1S 问题陈述模板
维度填写要点
What is it?关联 Jira / Bug ID；客观描述现象、发生前后过程、环境、操作。
How is it manifesting?如何复现？是否重复发生？有没有规律（与温度、时间、批次相关）？
Where are we seeing it?地点 + 发现人（FWQA / 量产线 / 客户端）。
When are we seeing it?首次发生时间，频率，是否与某次变更相关。
What is the consumer impact?用户感知（场景化描述），影响比例。
Impact Scope影响哪些项目 / 批次 / 机型，是否波及量产、其他产品线。
🧰 工具：HTDQ 框架
要素定义示例（机器人回充）
🦸 英雄 Hero试图解决问题的人 / 团队FW QA + 硬件团队
💎 宝藏 Treasure想要达成的目标所有机器都能稳定回充并满电
🐉 龙 Dragon阻碍达成目标的核心障碍2/3 EB1 样机 UI 一直显示充电中、实际未满
🎯 任务 Quest用"如何…"开头的开放式问题如何让 360 Fanjet EB1 稳定满电？
🧰 工具：框架完善四原则
  • 🎒 背包原则：只保留必须的内容，去掉冗余的环境描述。
  • 🐰 兔子原则：把所有相关信息都装进框架，不要遗漏关键变量（变更、批次、地区）。
  • 🐑 多利原则：复刻语言一致 — 同一概念全文用相同术语，不用同义词。
  • 🔍 华生原则：列出所有假设并标注，不要把假设当事实。
📤 输出

一份全面、准确、简洁、清晰的问题陈述（建议 1 页内）。包括：4W1H1S 表格 + HTDQ 一句话任务 + 已知假设清单。

⚠️ 常见陷阱
  • 把"问题"写成"方案"（"软件没有上报满电信号"是猜测，不是事实）。
  • 把"症状"当成"原因"（UI 一直显示充电中是症状，不是原因）。
  • 问题范围过宽（"机器人不工作"）或过窄（只描述某一台机器）。
2

分解问题 — 双向发散

⏱ 1~2 次 1.5h 会议 · 原则：发散思维，宁滥勿缺
💡 核心原则：这一阶段就是发散 — 想得越多越好，不评判可行性。后续阶段会有专门的收敛步骤。原则：所有问题都是由"变更"带来的。
🎯 行动
  1. 逆向分析（结果 → 原因）：从问题表现出发，多问几个 Why，列出所有可能导致此结果的因素。
  2. 正向分析（变更 → 结果）：列出问题机器相对历史正常机器有哪些差异和变更（按"人机物法环测"6 维度）。
  3. 把两条路径的因素汇总到一张大列表。
🧰 逆向分析工具组合
🐟 鱼骨图（因果图）
围绕问题画 6 大类骨架，每类下面再分支。最常用。
❓ 5 Whys
连续追问 5 次为什么，逼向深层原因。适合简单问题。
🌳 故障树分析（FTA）
用 AND/OR 逻辑门分解。适合产品质量、安全相关问题。
🧠 头脑风暴 / 书面头脑风暴
团队发散，书面版可保证匿名平等参与。
📊 流程图（Flowchart）
画出业务/工作过程，定位问题发生在哪个环节。
🧰 正向分析：人机物法环测 6M1E
维度检查范围
人 Man操作者、检验员、工艺员的身体状况、技能水平、工作责任心
机 Machine设备、工具、夹具：磨损、精度、校准状态
物 Material来料、元器件、原材料的批次、规格、供应商变更
法 Method工艺、操作方法、动作速度、安装次序、SOP
环 Environment照明、噪声、振动、温度、湿度、ESD
测 Measurement检验工具、测试方法、ATE/ICT 覆盖度

💡 实战技巧：对每个维度问"近期有什么变更？" — 新供应商、新设计、新软件版本、新人员、新工艺都是高嫌疑因素。

📋 模板：双向因素清单 
逆向（哪些原因可能带来这个问题）：
- 结构 1：材料透光率
- 结构 2：尺寸不匹配
- 电子 1：元器件 IR 灵敏度
- 软件 1：回充速度
- ...

正向（这次发现问题的机器有哪些差异/变更）：
- 结构 1：P 阶段是手板件
- 结构 2：dock 新设计改变角度
- 结构 3：更换边刷、滚刷电机
- 电子 1：新设计的 PCBA
- 电子 2：新的 IR 供应商
- 电子 3：新的电池
- 软件 1：修改了地图美化逻辑
- ...
📤 输出

一份结构化分解的影响因素清单，覆盖逆向 + 正向两条路径。

⚠️ 常见陷阱
  • 视野狭隘：只让懂这一模块的人参与 → 会议必须邀请跨模块人员。
  • 过早评判：发散阶段说"这个不可能" → 主持人要明确禁止打断。
  • 忽视变更管理：没去查最近的 ECN、版本提交记录、来料变化。
3

去掉非关键问题 — 收敛聚焦

⏱ 1 次 1h 会议 · 抓大放小，集中精力
💡 28 法则（帕累托原则）：20% 的因素往往导致 80% 的问题。这一步要识别这 20%，把精力集中过去。
🎯 行动
  1. 把上一阶段的因素清单逐条评估：相关性、嫌疑等级、验证成本。
  2. 剔除明显无关或低概率因素（要写明剔除理由，避免后续翻案争论）。
  3. 给保留下来的因素打优先级（P1 / P2 / P3）。
  4. 使用是-非矩阵区分"问题在哪发生 vs 在哪不发生"，缩小范围。
🧰 收敛工具
📊 帕累托图
如有历史数据，直接画图找出贡献最大的因素
✅ 是-非矩阵 (Is-Is Not)
列出问题"在哪 / 何时 / 何种条件下出现"vs"不出现"，差异点就是嫌疑因素
📋 矩阵图
多个原因 × 多个标准的二维评估矩阵
🗳️ 名义群组 / 配对比较
团队投票排序，确保民主达成一致
🕸️ 关系图
复杂情况下识别因素间的相互关系，找出"驱动因素"
📋 模板：等级清单
等级差异/变更具体因素剔除/保留理由
P1结构 1（P 阶段手板件）材料透光率样机和量产料差异最大，嫌疑高
P1电子 1（新设计 PCBA）R2529 阻值正好与充电检测电路相关
P2结构 2（dock 新设计）反射率需要测量数据验证
P3软件 2（回充速度）降速逻辑关联性较低，但易验证
—结构 3（更换边刷）—剔除：与回充无关
📤 输出

带优先级的关键因素清单，剔除项有书面理由（防止后续翻案）。

⚠️ 常见陷阱
  • 剔除得太狠，过早收敛 → 至少保留 P3 备选，留余地。
  • 剔除没有理由记录 → 一旦验证失败，会反复争论同一个因素。
  • 权威主导，不让基层发言 → 用名义群组技术保证平等投票。
4

制定工作计划 — PDCA 起点

⏱ 0.5 天 · 每个假设都要有责任人 + 验证方式 + 截止时间
💡 核心思想：PDCA = Plan（计划）→ Do（执行）→ Check（检查）→ Act（处理）。这一步是 Plan，可能需要多轮 PDCA 循环才能收敛。
🎯 行动
  1. 把每个保留因素拆成"待验证的假设"，用可证伪的语言描述。
  2. 为每个假设设计具体的验证方式（实验、对比测试、数据查询）。
  3. 分配责任人 + 完成时间 + 输出交付物。
  4. 提前约定下次会议时间，每次会议结束就开始倒计时。
🧰 数据收集与分析工具
📥 抽样
通过小样本推测总体（按批次、地区、机器型号分层）
📋 检查表
用预设表格系统化收集观测数据
📈 柱状图 / 帕累托图
识别数据模式或集中分布
📊 散点图
查看两个变量之间的相关性
🗺️ 问题集中图
在物理布局图上标记问题发生位置（如 PCB 板）
📋 模板：验证工作计划
等级差异/变更因素假设（可证伪）验证方式负责人完成时间
1结构 1材料透光率手板件透光率与量产件差异 > 5% 导致回充识别失败用分光仪测 5 个手板 vs 5 个量产件张三5/24
1电子 1R2529 阻值R2529 实测值低于设计 20K±1% 导致 MCU 误判过压拆 3 块问题板和 3 块正常板分别测阻李四5/26
2结构 2dock 反射率新 dock 角度变化导致 IR 反射偏离 ±10°用反射率测试治具对比新旧 dock王五5/28
📤 输出

带责任人和完成时间的验证工作计划，下次会议时间已预约。

⚠️ 常见陷阱
  • 假设写得太模糊（"R2529 不对"）→ 必须可证伪、可量化。
  • 没有截止时间或时间太宽 → 一周原则：每个验证项不超过 5 个工作日。
  • 责任人挂名不真做 → 责任人主管要在场签字承诺。
5

进行关键分析 — 假设验证

⏱ 1~3 周 · 假设为前提，事实为依据
🎯 核心准则：所有结论都要有事实证据支撑（数据、照片、日志、波形）。"凭经验"、"应该是"不是证据。
🎯 行动
  1. 按计划执行验证实验，记录原始数据和过程。
  2. 每个假设输出明确的"成立 / 不成立 / 部分成立"结论 + 证据链。
  3. 不成立的假设也要保留记录（防止后续重复）。
  4. 如有意外发现新的可疑因素 → 回到阶段 2-3-4 补充验证（PDCA 第 2 轮）。
  5. 每周开 30 分钟同步会议，刷新进度看板。
🧰 实战验证套路
  • 对比实验：好品 vs 坏品，新批次 vs 老批次，用同一条件。
  • 变量隔离：一次只改一个变量，避免混杂。
  • 复现验证：能稳定复现 = 接近根因；偶现 = 还要继续找。
  • 逆向验证：把怀疑的根因恢复成正常状态，问题是否消失。
  • 批量数据回查：用 ATE / BDP / 测试日志在历史数据中验证假设。
📋 模板：验证结果记录
因素验证方式结果证据结论
R2529 阻值 3 块问题板 vs 3 块正常板分别测阻（板上 + 拆下） 问题板拆下测得 19.6K，设计 20K±1%（即 19.8K~20.2K） 万用表实测照片 + 数据表 ✅ 成立 — 阻值偏低导致 MCU 误判过压
材料透光率 5 手板 vs 5 量产件分光仪测试 差异 < 1%，在公差内 分光仪报告 ❌ 不成立
📤 输出

每个假设的验证结果 + 可证实的因果关系链。

⚠️ 常见陷阱
  • 样本太少：3 块板可能不够，必要时扩大到 10-30 块。
  • 停在第一层原因：找到 R2529 阻值不对就结束 → 还要问"为什么阻值会不对？"（混料？测量精度？质量管控？）
  • 把相关当因果：散点图相关不等于因果，要做隔离实验。
  • 验证完不更新假设清单：要把已证伪的因素剔除，把新发现的加进去。
6

综合分析结果 — 构建论证

⏱ 0.5~1 天 · 用金字塔结构表达根因
🎯 行动
  1. 把所有验证结果整合，画出因果链：直接原因 → 中间原因 → 根本原因。
  2. 用金字塔结构组织论据：结论先行，论据相互支撑。
  3. 用"5 Whys"再深挖一次，确保已经触底（达到组织有能力消除的层级）。
  4. 把根因结论翻译成"如果消除 X，问题不再发生"的可证伪命题。
🧰 金字塔结构示例 
【顶层结论】机器人无法满电的根本原因：
  PCBA 上 R2529 电阻值低于设计规格，导致 MCU 误判电池过压

【第一层论据】
  ├─ 直接原因：R2529 实测 19.6K，设计 20K±1%
  │   └─ 证据：3 块问题板拆装前后测量数据
  ├─ 中间原因：来料阻值偏离规格但未被现有 ATE/ICT 测量精度检出
  │   └─ 证据：FAI 报告标注 R2529 正常，但中途换料未被拦截
  └─ 根本原因：质量管控流程对小阻值偏差的灵敏度不足
      └─ 证据：测量精度 vs 设计公差对比分析

【第二层论据 — 验证不成立的假设（剔除项）】
  ❌ 材料透光率（差异 < 1%）
  ❌ dock 反射率（在规格内）
  ❌ 边刷/滚刷电机变更（与回充无关）
  ...
🧰 5 Whys 深挖示范 
问：为什么机器无法满电？
答：因为 MCU 误判电池过压，触发保护停止充电。
问：为什么 MCU 误判过压？
答：因为采集到的充电电压偏小。
问：为什么采集电压偏小？
答：因为 R2529 阻值（19.6K）低于设计（20K）。
问：为什么 R2529 阻值会偏低？
答：因为中途换了不同批次的电阻，与设计规格 20K±1% 不符。
问：为什么换料没被发现？
答：因为现有 ATE/ICT 的测量精度不足以拦截这种偏差。  ← 这才是组织能消除的根因
📤 输出

清晰、简明、结构化的根本原因结论 + 论据链（一页内可呈现）。

⚠️ 常见陷阱
  • 停在直接原因，没追到根因（"R2529 阻值不对"是直接原因，"质量管控不足"才是根因）。
  • 追得太深超出组织能消除的层级（追到"供应商 CEO 重视不够"就脱离实际了）。
  • 论据散乱无层级，听众抓不住重点 → 必须用金字塔结构整理。
7

制定纠正与预防措施

⏱ 1~2 天 · Short term 救火 + Long term 治本
💡 措施分两类：Short term（针对直接原因或已发生影响的临时纠正）、Long term（针对根因的预防措施，避免重复发生）。两者都要给出。
🎯 行动
  1. 判断措施由本团队制定还是其他部门负责（如 SQE / 供应商 / 设计部门）。
  2. 列出 Short term 临时措施：拦截已影响范围、紧急修复。
  3. 列出 Long term 预防措施：改流程、改设计、加测试覆盖。
  4. 用加权标准评估法从多个候选方案中选优（成本、时间、有效性、风险）。
  5. 用力场分析识别推动和阻碍因素，提前应对阻碍。
  6. 用六项思考帽从不同视角检验方案完整性。
🧰 工具组合
⚖️ 加权标准评估
为成本/时间/有效性/风险分配权重，对每个方案打分
⚡ 力场分析
列出推动力 vs 阻力，量化强弱，制定削弱阻力策略
🎩 六项思考帽
事实/情感/批判/乐观/创造/管控 6 个角度审视
🌳 树图
把方案分解为可执行任务的层级结构
⚙️ TRIZ / SIT
面对工程矛盾时的创新发明原理

📤 输出

分类清晰、责任到人、有验收标准的措施清单。如果措施由其他部门给出，则只输出根因结论给到对方部门，进入下一阶段。

⚠️ 常见陷阱
  • 只有 Short term 没有 Long term → 短期问题解决了但根因还在，下次同样问题再来一次。
  • 措施虚化（"加强培训"、"提高重视"）→ 必须可观测、可验收。
  • 没有验收标准 → 跟踪时无法判断是否真的有效。
  • 跨部门措施没和对方负责人确认 → 形成空头支票。
8

汇总资料 — One Page 报告

⏱ 0.5 天 · 一页讲清楚事件、根因、措施
💡 目标：让没参与分析的领导/同事在 1 分钟内看懂"出了什么问题、为什么、怎么改"。
🧰 One Page 报告结构 
【标题区】Robotics 2026 | xxxxx 一句话摘要

【Key Takeaway】 2-3 行核心结论
- 根因一句话总结
- 影响范围 / 紧急程度

【Describe the Issue（4W1H1S）】
- What is it?
- How is it manifesting?
- Where are we seeing it?
- When are we seeing it?
- What is the consumer impact?
- Impact Scope

【What is the root cause?】
根因结论 + 关键证据 + 因果链（建议配图）

【Short term】
- 措施 1（负责人 / 截止时间）
- 措施 2（负责人 / 截止时间）

【Long term】
- 措施 1（负责人 / 截止时间）
- 措施 2（负责人 / 截止时间）

【Notes / 备注】（可选）
🧰 表达原则
  • 结论先行：标题一句话讲清根因。
  • 用图代替文字：因果链、对比数据用可视化。
  • 少术语多场景：写给非专业人员能看懂。
  • 关键数据要附原始证据链接（Jira、Confluence、文件路径）。
📤 输出

One Page 报告（PDF 或 PPT 单页），存档至 Confluence / 内部 Wiki。

⚠️ 常见陷阱
  • 报告写成长文，超过 1 页 → 强制单页约束。
  • 只讲技术细节不讲影响 → 要让管理层看到业务影响。
  • 没有附证据链接 → 后期复盘找不到原始资料。
9

跟踪验证与闭环

⏱ 持续监控 1~3 个月 · 关闭 / 重启
🎯 核心准则：没有"验证有效"的环节，RCCA 就不算完成。每条措施都要被检查是否真的解决了问题。
🎯 行动
  1. 按措施清单逐项跟踪完成情况，更新到 Jira / Confluence。
  2. 对每条 Long term 措施做有效性验证：在新批次/新版本上确认问题不再发生。
  3. 如果问题再次出现：重启 RCCA，回到阶段 1 重新审视问题定义。
  4. 关闭 Jira：所有措施完成 + 至少一个监控周期无再发。
  5. 做过程复盘：分析这次 RCCA 哪里高效、哪里低效，沉淀经验。
  6. 归档：Wiki 文档 + 案例库 + 模板更新。
🧰 跟踪工具
  • 📋 措施跟踪表（Jira Epic / Confluence 页面 / Excel）
  • 📊 监控仪表盘：用同一类问题的发生率 / 失效率作为有效性指标
  • 📝 复盘会议（Retrospective）：1 小时，讨论"做得好 / 做得不好 / 下次改进"
  • 📚 经验库：把根因模式抽象成"问题模板"，下次类似问题直接套用

📤 输出

关闭的 Jira + Wiki 归档 + 经验沉淀。

⚠️ 常见陷阱
  • "措施做完就完事"，没有验证有效 → 下次同样问题还会复发。
  • 没有复盘 → 同样的 RCCA 错误每次都犯。
  • 不更新模板和案例库 → 组织能力没有积累。

🧭 贯穿全程的思维模式

RCCA 不只是流程，更是一组思维习惯。以下 6 项思维方式贯穿全部 10 个阶段。

🐢 第二系统思维（慢思考）

对抗本能反应，用证据驱动决策。"听起来合理"不等于"真的对"。永远问：我有什么证据？

🎲 概率思维

不追求 100% 确定，承认不确定性。用"嫌疑度"而非"是/否"评估每个假设。

🔄 迭代精神

把每个分析当成一次实验，欢迎被推翻。多轮 PDCA 比一次完美更可靠。

🤝 团队协作

一人盲点多。鼓励争论但不对人，营造信赖、开放、诚实的氛围。

📊 证据驱动

所有结论都有数据、日志、照片支撑。"以前类似问题是 X"不是证据，是经验偏见。

🛡️ 预防优于治疗

FMEA / FMECA / DRBFM 提前识别风险。每次 RCCA 都把根因变成下一次的预防点。
